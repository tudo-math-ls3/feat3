Detailed Description

template<typename MatrixA_, typename MatrixB_, typename MatrixD_, typename Filter_>
class FEAT::Solver::Vanka< LAFEM::SaddlePointMatrix< MatrixA_, MatrixB_, MatrixD_ >, Filter_ >

Vanka preconditioner/smoother class template implementation.

This class template implements the Vanka preconditioner/smoother, which is a special Block-SOR/Gauss-Seidel variant suitable for working with saddle-point systems of the form

\[\begin{bmatrix} A & B\\D & 0\end{bmatrix} \cdot \begin{bmatrix} v\\p\end{bmatrix} = \begin{bmatrix} f\\g\end{bmatrix}\]

In total, this class template implements eight different variants of the Vanka, which can be chosen at run-time by specifying the corresponding VankaType in the constructor of this object. The eight variants are the cross-product of three binary variants:

1.a: Block Variants (VankaType::block_*_*)
The block variants collect all pressure DOFs, which are adjacent to each other via the connectivity graph of D*B with the same degree. If the B and D matrices are build by using a discontiunuous pressure element, then these block variants collect all DOFs associated with a single element of the underlying mesh, thus leading to the 'true' element-based Vanka.

  • Block variants are recommended for discontinuous pressure spaces.

1.b: Nodal Variants (VankaType::nodal_*_*)
The nodal variants process exactly one pressure DOF per block.

  • Nodal variants are recommended for continuous pressure spaces.

2.a: Diagonal Variants (VankaType::*_diag_*)
These variants apply the Schur-complement approach onto each local system and approximate the inverse of A by the inverse of its main diagonal. If there is more than one local pressure DOF, the local Schur-Complement matrix itself is inverted by using Gaussian elimination.

2.b: Full Variants (VankaType::*_full_*)
These variants invert the full dense local systems by using Gaussian elimination.

3.a: Multiplicative Variants (VankaType::*_*_mult)
These variants work successively within one Vanka iteration, just as the (scalar) Gauss-Seidel- or SOR-iterations do.

3.b: Additive Variants (VankaType::*_*_add)
These variants work in an additive way, i.e. they ignore the 'progess' made in the current iteration, just as the (scalar) Jacobi-iterations do.

Moreover, each of the above variants is parameterized in a relaxation/damping parameter omega, which can be specified in the constructor. For multiplicative variants, omega is a relaxation parameter (as in SOR), whereas for additive variants omega is a damping parameter (as in damped Jacobi).

Guidelines, Remarks, Warnings & Tips

  • In many cases, setting the relaxation/damping parameter omega to a value less than 1 can 'cure' a diverging Vanka variant. So before trying out a more expensive variant to achieve convergence, you may want to play around with omega first, e.g. by setting it to 0.8.
  • In many cases and for any variant, choosing omega > 1 leads to divergence, so don't do that.
  • If the pressure space is discontinuous, one should always choose block variants instead of nodal variants, as these require less memory, are less computationally expensive and lead to faster convergence in most cases.
  • If the pressure space is continuous, one should stick to the nodal variants.
    Moreover, never use full variants for continuous pressure spaces, as these may require astronomical amounts of memory for the factorization!
  • Diagonal variants usually require (significantly) less memory than full variants. Therefore, when trying to configure a Vanka for a new type of problem and/or discretization, one should first try whether a diagonal variant works before trying a full variant.
  • For more complex problems (like convection-dominant Navier-Stokes), the full variants are often more stable than the diagonal variants.
  • Multiplicative variants are roughly twice as computationally expensive as additive variants.
  • Surprisingly, additive variants seem to converge faster than their multiplicative counterparts in many cases (at least for linear Stokes equations).
